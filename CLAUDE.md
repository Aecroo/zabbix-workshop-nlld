# CLAUDE.md

This file provides guidance to Claude Code when working with the Zabbix Berlin Conference 2026 Workshop project.

## Project Overview

A Go-based mock data API for workshop attendees to interact with and gather realistic sensor data for testing **Zabbix 7.4 nested low-level discovery (LLD) features**. The API simulates a demo company building with rooms and various sensors, generating gradually-changing values suitable for graphing.

## ID System

IDs are **autogenerated** based on the order of appearance in the YAML configuration:

| Entity | ID Range | Assignment |
|--------|----------|------------|
| Buildings | 10-99 | Sequential by config order (first = 10) |
| Rooms | 100-999 | Sequential across all buildings (first = 100) |
| Sensors | 1000-9999 | Sequential within each room (first = 1000) |

**IDs are persistent**: The same configuration always produces the same IDs. Do not specify `id` fields in YAML config - they are ignored.

## API Endpoints

| Endpoint | Description |
|----------|-------------|
| `GET /api/all` | Get all buildings, rooms, sensors, and readings in one request (nested array structure) |
| `GET /api/buildings/` | List all buildings/locations |
| `GET /api/buildings/{id}/` | Get building details by ID |
| `GET /api/buildings/{id}/rooms/` | List rooms in a building (minimal, no sensor_ids) |
| `GET /api/rooms/{id}/` | Get room details by ID (minimal, no sensor_ids) |
| `GET /api/rooms/{id}/sensors/` | List sensors in a room |
| `GET /api/sensors/{id}/readings/` | Get latest sensor readings (fused reading) |
| `GET /swagger/` | OpenAPI Swagger UI documentation |
| `GET /health` | Health check endpoint |

**Note**: The `/api/all` endpoint returns a nested array structure: buildings → rooms (array) → sensors (array, each with readings nested). This is ideal for Zabbix LLD testing as it provides the complete hierarchy in one request.

## Chatty REST Architecture

The API follows a **chatty REST pattern** where individual endpoints return minimal data to encourage proper resource navigation:

- `GET /api/buildings/{id}/rooms/` returns rooms **without** sensor_ids
- `GET /api/rooms/{id}/` returns room details **without** sensor_ids
- To get sensors, you must call `GET /api/rooms/{id}/sensors/` separately
- Only `/api/all` provides the complete nested hierarchy in one request

This design:
- Encourages proper REST resource navigation patterns
- Reduces payload size for individual endpoint calls
- Mirrors real-world API behavior where relationships are often separate endpoints
- Provides a realistic testing scenario for Zabbix LLD with multiple HTTP requests

## Configuration

The API loads mock data from `config/default.yaml`. The YAML structure defines buildings with nested rooms and sensors:
- **Sensor types**: `environment` (fused temp/humidity/CO2) or `power` (fused voltage/current/power/energy)
- **Custom config**: Use `CONFIG_PATH=/path/to/config.yaml` to override the default
- **Fallback chain**: CONFIG_PATH env → config/default.yaml → embedded default.yaml

**Important**: Do not specify `id` fields in YAML - they are autogenerated based on order of appearance.

## Key Implementation Details

1. **Autogen ID System**: Buildings start at 10, rooms at 100, sensors at 1000 - assigned sequentially based on config order in `internal/generators/data.go`

2. **Fused Sensor Readings**: Each sensor returns all its metrics in a single reading object:
   - Environment sensors return `EnvironmentReading{sensor_id, temperature, humidity, co2, timestamp}`
   - Power sensors return `MultiReading{sensor_id, voltage, current, power, energy, timestamp}`

3. **Random Walk Algorithm**: Sensor values change gradually every 5+ seconds with configurable noise factors:
   - Temperature: ±0.5°C per update
   - Humidity: ±3% per update
   - CO2: ±80ppm per update
   - Power: ±10W base variation

4. **Time-of-Day Patterns**: Power sensors vary based on hour (in `internal/generators/sensors.go`):
   - Business hours (9-18): 70-100% load
   - Morning ramp-up (6-9): 50-70% load
   - Evening wind-down (18-22): 40-60% load
   - Night standby: 20-30% load

5. **Room-Type Baselines**: Detected by room `type` field, not ID:
   - Server rooms (`type: serverroom`): cooler (17-24°C), lower humidity (35-55%)
   - Kitchens (`type: kitchen`): warmer (21-30°C), higher humidity baseline (60%)
   - Training rooms (`type: training_room`): higher CO2 baselines (up to 2500ppm)

6. **No Authentication**: Open API for easy workshop access with CORS enabled (`*` origin)

7. **Thread-Safe State Management**: Each sensor has its own RWMutex for concurrent read/update safety

## Architecture Notes

The project uses a standard Go web application structure:
- `cmd/` contains the main entry points
- `internal/` holds private application code not importable by external packages
- `pkg/` exports reusable packages

**Key patterns:**
- Embedded files for Swagger UI (openapi.json, ui.html) and fallback config using `//go:embed`
- Thread-safe sensor state management with RWMutex per sensor
- Configuration validation on load (missing required fields, invalid types)
- Standard API response wrapper: `{success, data?, error?, meta?}`
- Minimal room responses (`RoomMinimal`) for chatty REST pattern

**Key files to understand:**
- `internal/handlers/api.go` - All HTTP handlers with response formatting
- `internal/generators/data.go` - Config loading and ID autogeneration
- `internal/generators/sensors.go` - Random walk algorithm and time-of-day patterns
- `internal/models/types.go` - Runtime data structures including nested types for `/api/all`
- `internal/models/config_types.go` - YAML config types with conversion functions

When adding new endpoints or features, follow this pattern and maintain the resource hierarchy (buildings → rooms → sensors).
